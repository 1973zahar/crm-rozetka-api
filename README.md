# Marketplace CRM

## Mandatory Project Logic

```text
This repo is one autonomous block in an Odoo-like modular CRM.
It must interact with the main PostgreSQL/SQL core for shared data and exchange.
Every chat created for this repo must obey D:\Codex\CRM\AGENTS.md, this repo's AGENTS.md, and D:\Codex\CRM\docs\PROJECT_SHARED_RULES.md before doing project work.
Every chat and block builder must read and update D:\Codex\CRM\docs\PROJECT_SHARED_KNOWLEDGE_BASE.md for durable cross-block facts.
```

Ця версія CRM працює з маркетплейсами та B2B контуром. Перші екрани ведуть оператора через публікації, імпорт товарів, імпорт замовлень, статуси, ТТН, оплати, аналітику, B2B заявки, відповідальне зберігання й кабінети клієнтів.

У навігації активні модулі: `Панель`, `Продажі`, `B2B`, `Товари`, `Аналітика та звіти`, `Фінанси`, `Налаштування`.

Rozetka API працює тільки на завантаження з Rozetka в CRM. Дані тягнуться з `Налаштування → 6.1 Маркетплейси · Rozetka API → CRM`; замовлення після імпорту видно в `Продажі → 2.1 Замовлення`, товари та публікації — у `Товари`.

B2B кабінети, відповідальне зберігання, заявки на відвантаження та звіти продажу клієнтом винесені в окремий головний модуль `B2B`. Ширші сценарії приходу, серійного обліку й повного магазину можуть лишатися підтримуючою логікою або окремими підпунктами.

# Rozetka API inbound

У цій окремій версії CRM додано read-only блок `Rozetka API · отримання товарів і замовлень` у розділі `Маркетплейси`.

Перед запуском скопіюйте `.env.example` у `.env` і заповніть один із варіантів:

```ini
ROZETKA_API_TOKEN=ваш_токен
```

або:

```ini
ROZETKA_USERNAME=ваш_логін
ROZETKA_PASSWORD=ваш_пароль
```

Після цього запускайте CRM звичайно через `start-crm.bat`. Backend `mock-api.ps1` проксуватиме:

- `/api/rozetka/items/search`;
- `/api/rozetka/items/search/all`;
- `/api/rozetka/image`;
- `/api/rozetka/goods/new`;
- `/api/rozetka/goods/new/all`;
- `/api/rozetka/orders/search`;
- `/api/rozetka/orders/import`;
- `/api/rozetka/orders/{id}`.

Товари Rozetka імпортуються в каталог CRM і публікації Rozetka. Основний режим товарів використовує `/items/search`, бо він повертає повний активний каталог. `/goods/new` лишився як окремий режим для нових/неопрацьованих позицій. Замовлення Rozetka імпортуються в журнал замовлень маркетплейсів. Повні сирі поля Rozetka зберігаються у `rozetka.raw` відповідної картки.

Фото Rozetka показуються через локальний endpoint `/api/rozetka/image?url=...`, щоб браузер не ламав прямі зовнішні посилання.

Для HTTPS-запитів до Rozetka backend використовує маленький Node.js-помічник `src/rozetka-http.mjs`, бо Windows PowerShell може падати на TLS-з'єднанні. Якщо Node.js не доступний у `PATH`, вкажіть повний шлях у `.env`:

```ini
ROZETKA_NODE_PATH=C:\Users\User\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe
```

# Arms CRM MVP

Операційний прототип CRM для бізнесу з продажу цивільної зброї та аксесуарів. Система покриває магазин, сайт, маркетплейси, B2B продажі, відповідальне зберігання, складський облік, фінанси, ролі, звіти та підготовку до двосторонніх інтеграцій з BAS/BAF, банками й маркетплейсами.

Поточна версія є локальним frontend/MVP: дані зберігаються у браузері в `localStorage`, а mock API запускається через PowerShell. Це не production-сховище, а робочий прототип логіки та інтерфейсів.

## Облік залишків, фірм і цін

Залишки ведуться не тільки по товару, а по комбінації `товар + фірма + склад + клієнт B2B`. Це означає, що товар, який прийшов на ФОП, не може бути списаний у продажі від ТОВ, і навпаки. Для звичайних товарів система перед проведенням продажу, маркетплейс-накладної, B2B-передачі або B2B-звіту перевіряє достатній залишок саме по вибраній фірмі та складу. Для зброї перевіряється серійний номер, модель, склад, фірма-власник, актуальність, статус продажу та ЄРЗ.

У проведених накладних фірма не змінюється вручну, бо вона вже прив'язана до складського руху. Якщо потрібна зміна фірми, складу або товарних рядків після проведення, це робиться коригувальним документом.

У розділі `Склад` інвентаризація та залишки мають окремий відбір по складу і по фірмі. Підсумкові картки, таблиця та інвентаризаційний звіт рахуються за вибраним відбором. Курси валют редагуються в `Налаштуваннях`, а не в складському розділі.

Грошові суми відображаються з копійками/центами для всіх валют. Конвертація йде через налаштування курсів: якщо задано `UAH = 45`, `USD = 1`, `EUR = 0.85`, то `0.24 USD` у гривні дорівнює `10.80 грн`, а накладна з рядками `0.24 USD + 5 400 грн` показує `5 410.80 грн`, без округлення до цілих.

## Швидкий запуск

Локально на одному комп'ютері:

```powershell
.\start-crm.bat
```

Або напряму:

```powershell
powershell -ExecutionPolicy Bypass -File .\mock-api.ps1 -Port 8789
```

Відкрити:

```text
http://127.0.0.1:8789/index.html
```

Для тестування з інших комп'ютерів у локальній мережі використовуйте `start-crm-lan.bat` і відкривайте адресу комп'ютера-сервера з портом `8789`.

## Вхід і користувачі

Перед відкриттям CRM показується екран входу.

Demo-входи працівників:

```text
Адміністратор: admin / admin
Керівник маркетплейсів: lead / 1234
Менеджер маркетплейсів: manager / 1234
Оператор складу: sklad / 1234
Логіст: logist / 1234
Аналітик маркетплейсів: analyst / 1234
```

Після входу не можна перемикати користувача або роль у верхній панелі. Для зміни користувача потрібно натиснути `Змінити користувача` і зайти під іншим логіном та паролем.

Адміністратор у розділі `Ролі` може:

- створювати працівників;
- редагувати ПІБ, відділ, телефон, email;
- задавати логін і пароль;
- змінювати роль;
- вмикати або вимикати працівника;
- змінювати права доступу для всіх ролей.

## LAN-тестування

Для тестування кількома людьми у локальній мережі запускайте `start-crm-lan.bat` на комп'ютері-сервері. Скрипт слухає `0.0.0.0:8789`, показує IPv4-адреси цього комп'ютера, а тестери в тій самій Wi-Fi/LAN мережі відкривають:

```text
http://IP_КОМПЮТЕРА_СЕРВЕРА:8789/index.html
```

Якщо Windows Firewall запитає дозвіл, потрібно дозволити доступ для приватної мережі. Для реального інтернет-доступу потрібен сервер із базою даних, HTTPS і окремим backend API, бо поточний MVP зберігає дані у `localStorage` браузера.

## Ролі та права доступу

Розділ `Ролі` містить матриці прав:

- базові права ролі;
- доступ до модулів CRM;
- дозволи на створення документів;
- дозволи на зміну проведених документів;
- дозволи на редагування окремих полів документів.

Типові ролі у demo-версії: `Адміністратор`, `Керівник продажів`, `Менеджер магазину`, `Зав склад`, `Логіст`, `B2B клієнт`.

Основні базові права:

- зміна закритого дня;
- продаж зброї;
- зміна цін;
- експорт/облік BAS/BAF;
- погодження кредиту та відтермінування;
- керування працівниками;
- перегляд звітів;
- зміна налаштувань;
- друк документів і звітів.

У журналах документів і звітах є період `Дата від / Дата до`. Кнопка `Друк` активна тільки для ролей, яким адміністратор дозволив право `Друк документів і звітів`.

Права на документи охоплюють:

- продаж / накладну;
- прихід;
- відповідальне зберігання;
- звіт продажу B2B;
- оплату;
- витрату;
- кредиторку;
- картку товару;
- картку клієнта;
- серійний облік;
- публікацію маркетплейсу;
- замовлення маркетплейсу;
- зміну статусу маркетплейс-замовлення;
- відстеження доставки / ТТН;
- відстеження оплати маркетплейсу;
- налаштування.

Права на поля охоплюють:

- дату документа;
- клієнта;
- товар / QR;
- склад;
- ціну;
- знижку;
- собівартість;
- серійні номери;
- дозвіл покупця;
- оплату / курс;
- відтермінування;
- менеджера;
- бухоблік BAS/BAF;
- маркетплейс;
- статус маркетплейс-замовлення;
- ТТН / статус доставки;
- статус оплати маркетплейсу.

Якщо поле заборонене для ролі, система блокує його в інтерфейсі. Якщо документ заборонений для ролі, створення документа блокується.

Окремий блок `Дозволи на зміну проведених документів` керує редагуванням документів після проведення або закриття дня. Адміністратор може видати це право по кожному виду документа окремо для кожної ролі:

- проведена накладна продажу;
- проведений прихід;
- проведене відповідальне зберігання;
- проведений звіт продажу B2B;
- проведена оплата;
- проведена витрата;
- проведена кредиторка;
- проведене замовлення маркетплейсу;
- проведений серійний документ.

У журналах такі права вмикають кнопку `Змінити`. Для товарних документів система дозволяє змінювати реквізити документа, менеджера, дати, доставку, оплатні та бухгалтерські ознаки. Товарні рядки, кількість, серійні номери та складський рух змінюються через коригування, щоб не зламати залишки, серії та BAS/BAF обмін.

`Зав склад` має доступ до складських операцій, серійного обліку, передачі маркетплейс-замовлень на склад і друку. `Логіст` має доступ до маркетплейс-замовлень, передачі на доставку та API-відстеження ТТН. Підтягування оплат маркетплейсу або поштового сервісу обмежується ролями з правом `Відстеження оплати`.

## Типи товарів

Система підтримує два типи товарів.

### Зброя

Обов'язкові реквізити:

- модель;
- калібр;
- бренд;
- QR або штрихкод;
- артикул постачальника;
- внутрішній код моделі;
- код УКТЗЕД;
- ознака перевірки ЄРЗ;
- облік кожної одиниці за унікальним серійним номером.

Правила для зброї:

- прихід зброї не проводиться без серій;
- кількість серій має дорівнювати кількості в приході;
- дублювання серій заборонене;
- продаж зброї без серії заборонений;
- продаж серії іншої моделі заборонений;
- повторний продаж уже проданої серії заборонений;
- для продажу потрібні номер дозволу покупця і дата видачі дозволу;
- для продажу потрібна перевірка ЄРЗ;
- серії, які продані або неактуальні, показуються як недоступні.

### Звичайні товари

Обов'язкові реквізити:

- модель;
- бренд;
- QR або штрихкод;
- артикул постачальника;
- внутрішній код моделі;
- код УКТЗЕД.

Звичайні товари ведуться кількісно, без серійного обліку.

## Картка товару

У створенні товару поля довідників мають випадаючі списки з можливістю додати нове значення:

- категорія;
- одиниця;
- бренд;
- модель;
- калібр;
- УКТЗЕД;
- артикул постачальника;
- внутрішній код.

QR або штрихкод є обов'язковим. Його можна внести вручну або згенерувати.

Фото товару:

- до 6 фото;
- підтримуються JPG/JPEG та PNG;
- фото приводяться до формату, придатного для маркетплейсів;
- фото використовуються у публікаціях маркетплейсів.

Ціни, собівартість, оплати, курси та інші грошові поля приймають дробові значення як через кому, так і через крапку: `4,8`, `4.8`, `1234,56`.

Прайси товарів:

- `Прихідна` ведеться як собівартість і валюта приходу.
- Активні прайси продажу, наприклад `Роздрібна`, `B2B`, `B2B дилер`, `Маркетплейс`, показуються окремими полями в картці товару.
- У картці клієнта вибирається прайс продажу, і продаж/B2B підтягує ціну саме з цього прайса.
- У `Налаштуваннях` адміністратор може додавати нові прайси або вимикати непотрібні.
- У ролях право `Ціна` обмежує зміну продажних цін, а право `Собівартість` обмежує зміну прихідної ціни.

## Довідники документів

У `Налаштуваннях` є блок `Довідники документів`. Адміністратор може створювати варіанти для полів, які мають випадаючий список, і задавати ієрархію через батьківський пункт.

Довідники використовуються в документах для:

- типів документів продажу;
- типів документів приходу;
- каналів продажу;
- служб доставки;
- платників доставки;
- режимів оплати;
- джерел оплати;
- умов оплат клієнтів;
- типів клієнтів;
- податкових режимів;
- методів витрат;
- джерел B2B-звіту;
- статей руху коштів;
- статей витрат;
- типів складів.

Приклад ієрархії: `Доставка / Нова пошта / Кур'єр`. У документах такий пункт з'являється у відповідному списку вибору.

## Прихід товару

Прихід використовується для внесення товару на склад.

Форма приходу побудована аналогічно продажам: у шапці документа вказуються тип документа, дата, фірма, склад, постачальник, документ постачальника та ознака BAS/BAF, а товари вносяться рядками. Кнопка `Додати ще товар` додає нову позицію в той самий документ; для кожного рядка окремо задаються QR/штрихкод, товар, кількість, прихідна ціна, валюта, ЄРЗ, актуальність і серійні номери.

Основні поля:

- дата;
- постачальник;
- можливість створити нового постачальника;
- документ постачальника;
- фірма;
- склад;
- товар;
- QR / штрихкод;
- кількість;
- ціна приходу;
- валюта приходу;
- серійні номери для зброї;
- ЄРЗ статус;
- ознака BAS/BAF.

Логіка приходу зброї:

1. Менеджер вибирає товар типу `Зброя`.
2. Система перевіряє обов'язкові реквізити картки товару.
3. Менеджер вводить серійні номери.
4. Кількість серій має точно дорівнювати кількості приходу.
5. Система блокує дублікати в документі.
6. Система блокує серії, які вже існують у CRM.
7. Після проведення створюються окремі серійні одиниці.
8. Серії потрапляють на вибраний склад.
9. Документ можна експортувати або позначити для BAS/BAF.

Логіка приходу звичайних товарів:

1. Менеджер вибирає товар без серійного обліку.
2. Система перевіряє QR або штрихкод.
3. Кількість додається до складського залишку.
4. Собівартість і валюта приходу оновлюють облікові дані товару.

## Продажі

Канали продажу беруться з довідника `Канали продажу` у налаштуваннях:

- магазин;
- сайт;
- Rozetka;
- Prom;
- Epicentr;
- Allo;
- B2B.

Нова накладна створюється у табличному режимі. Реквізити шапки документа розміщені в одному рядковому блоці, а товари вносяться окремими рядками. Кнопка `Додати ще товар` додає наступний рядок без втрати вже введених даних.

Поля шапки продажу:

- тип документа;
- дата;
- договір або підстава;
- склад списання;
- фірма;
- канал;
- клієнт;
- прайс;
- доставка;
- ТТН;
- платник доставки;
- стаття руху коштів;

- відтермінування;
- менеджер;
- ознака BAS/BAF;
- коментар.

Поля рядка товару:

- QR / штрихкод;
- товар;
- кількість;
- ціна;
- валюта;
- знижка;
- серійні номери;
- номер дозволу;
- дата видачі дозволу.

Логіка продажу зброї:

1. Менеджер вибирає товар у рядку накладної.
2. Система показує тільки серії, прив'язані до моделі цього рядка.
3. Серії іншої зброї не приймаються.
4. Продані, неактуальні або неперевірені ЄРЗ серії блокуються.
5. Кількість у рядку має дорівнювати кількості вибраних серій.
6. Обов'язково вводиться номер дозволу покупця і дата видачі у цьому рядку.
7. Після проведення серія отримує статус `sold`.
8. Повторний продаж цієї серії блокується.

Логіка продажу звичайного товару:

1. Система перевіряє QR або штрихкод товару.
2. Кількість списується зі складу.
3. Створюється накладна.
4. Якщо оплата внесена одразу, створюється платіж.
5. Якщо є відтермінування, створюється дебіторка до дати оплати.

Якщо в накладній рядки у різних валютах, підсумок документа перераховується у валюту обліку за курсами з налаштувань.

## B2B і відповідальне зберігання

Відповідальне зберігання означає, що товар фізично відвантажений на склад клієнта, але власність залишається нашою до моменту продажу товару клієнтом.

### Передача на відповідальне зберігання

Поля документа:

- дата передачі;
- B2B клієнт;
- QR / штрихкод;
- товар;
- кількість;
- термін оплати після продажу;
- серійні номери для зброї;
- відповідальний менеджер;
- коментар.

Логіка:

1. Менеджер вибирає B2B клієнта.
2. Система визначає або створює склад клієнта.
3. Для звичайного товару кількість списується з нашого складу і додається на склад клієнта.
4. Для зброї вибрані серії переводяться на склад клієнта.
5. Серії залишаються у статусі відповідального зберігання.
6. Власність не переходить до клієнта.
7. Створюється документ відповідального зберігання.

Для передачі на зберігання показуються вільні актуальні серії цієї моделі. Продаж зі зберігання все одно блокує серії без перевірки ЄРЗ.

### Продаж клієнтом зі складу відповідального зберігання

Клієнт може:

- самостійно провести продаж у своєму B2B кабінеті;
- передати звіт продажу менеджеру, а менеджер внесе продаж у кабінеті клієнта.

Поля звіту продажу:

- дата продажу;
- B2B клієнт;
- QR / штрихкод;
- товар зі складу клієнта;
- кількість;
- ціна продажу;
- валюта;
- термін оплати;
- джерело звіту;
- серійні номери для зброї;
- номер дозволу покупця;
- дата видачі дозволу;
- менеджер;
- коментар.

Логіка:

1. Система перевіряє залишок саме на складі вибраного клієнта.
2. Для зброї можна вибрати тільки серії цього клієнта і цієї моделі.
3. Продані або чужі серії блокуються.
4. Після проведення створюється накладна B2B.
5. Саме з цього моменту власність переходить до клієнта.
6. Виникає дебіторська заборгованість.
7. Дата оплати визначається за терміном, який ввів менеджер.
8. Залишок на відповідальному зберіганні зменшується.

### B2B кабінет клієнта

У кабінеті видно:

- що і коли відвантажено на відповідальне зберігання;
- кому відвантажено;
- скільки передано;
- скільки продано;
- залишок;
- серії зброї;
- QR / штрихкоди;
- оплати;
- дебіторку;
- інвентаризацію.

Клієнт може сформувати:

- звіт залишків;
- звіт оплат;
- інвентаризаційний звіт.

Інвентаризація:

- зброя перевіряється за серійними номерами;
- звичайні товари перевіряються за QR або штрихкодами.

## Маркетплейси

Підтримуються контури:

- Rozetka;
- Prom;
- Epicentr;
- Allo.

Модуль маркетплейсів містить:

- список публікацій;
- окремі SKU під кожен маркетплейс;
- назву публікації;
- зовнішній ID;
- ціну;
- валюту;
- фото;
- статус синхронізації;
- менеджера.

Блок `Створити публікацію` працює у горизонтальному табличному режимі. Один рядок дорівнює одній публікації конкретного маркетплейсу: маркетплейс, товар, SKU, зовнішній ID, назва, ціна, валюта та менеджер. Кнопка `Додати ще` додає наступний рядок без втрати вже введених даних, а система блокує дублювання SKU в межах одного маркетплейсу як у формі, так і серед уже створених публікацій. Блок `Обмін даними` винесений у самий низ сторінки маркетплейсів, щоб робота з публікаціями, списками та замовленнями не стискалась поруч із API-діями.

Замовлення маркетплейсів містять:

- дату створення, дату погодження, передачі на склад, передачі на доставку, вручення та оплати;
- маркетплейс і зовнішній номер замовлення;
- покупця та автоматично створеного клієнта CRM;
- товар, SKU, кількість і суму;
- службу доставки, ТТН, API-статус доставки та час останньої перевірки;
- джерело оплати, API-статус оплати та час останньої перевірки;
- кольоровий статус: `новий`, `погоджено`, `передано на склад`, `передано на доставку`, `доставлено / вручено`, `оплачено`.

У вкладці `Замовлення маркетплейсів` є комбінований відбір:

- за датою від / до;
- за статусом;
- за конкретним маркетплейсом;
- умови можна використовувати окремо або разом.

API-дії винесені у блок `Обмін даними`: там обирається замовлення, після чого можна перевірити ТТН через API доставки або підтягнути оплату через API маркетплейсу / поштового сервісу. В рядках таблиці залишаються тільки операційні дії: повідомити менеджера, створити клієнта, погодити, передати на склад, створити накладну, передати на доставку.

Логіка роботи з маркетплейсами:

1. Створюється або імпортується публікація.
2. До публікації прив'язується товар CRM.
3. Експортуються реквізити товару, фото, ціни і залишки.
4. Імпортуються оновлення публікацій.
5. Залишки можуть синхронізуватися з CRM.
6. Ціни можуть оновлюватися по курсу або вручну менеджером.
7. Замовлення імпортується в CRM.
8. Менеджер отримує подію про нове замовлення.
9. Покупець автоматично створюється як клієнт, якщо його ще немає.
10. Менеджер погоджує асортимент, доставку й оплату.
11. Замовлення передається на склад для комплектації.
12. Після створення накладної логіст передає замовлення на доставку; якщо ТТН ще немає, система створює demo-ТТН.
13. API доставки за ТТН оновлює статус до `доставлено / вручено`.
14. API маркетплейсу або поштового сервісу підтверджує оплату, створює фінансову оплату до накладної й переводить замовлення в `оплачено`.

Для зброї автоматична накладна з маркетплейсу блокується: потрібно вручну вибрати серії та внести дозвіл покупця.

## Фінанси

Фінансовий блок містить:

- платежі з прив'язкою до накладних;
- готівку;
- безготівку;
- банк/API як канал оплати;
- оплата карткою з вибором термінала;
- фірма зарахування для кожної оплати;
- ознака ПРРО для готівки;
- аванс клієнта без прив'язки до накладної;
- мультивалютність;
- курс дня;
- касові зміни;
- дебіторку;
- кредиторку;
- витрати.

Логіка оплат:

1. Спочатку вибирається джерело: готівкова каса, банк або картка.
2. Для картки обов'язково вибирається фірма зарахування і термінал з довідника налаштувань.
3. Для готівки вибирається, чи проводити операцію через ПРРО.
4. Оплату можна прив'язати до конкретної накладної або внести як аванс клієнта.
5. Сума може бути в різній валюті.
6. Система використовує курс дня або ручний курс у полі оплати.
7. Часткова оплата переводить накладну в статус `partial`.
8. Повна оплата переводить накладну в статус `paid`.
9. Неоплачена сума формує дебіторку.
10. Аванс зберігається як окремий платіж клієнта без зміни статусу накладних.

Курси в налаштуваннях задаються як еквівалент `1 USD`: наприклад `UAH = 45`, `USD = 1`, `EUR = 0,85`. У такій логіці оплата `420 USD` у накладній `EUR` зараховується як `420 * 0,85 = 357 EUR`, а не як ділення на `0,85`.

Дебіторка відображається:

- по клієнтах;
- по менеджерах;
- по накладних;
- у різних валютах через курс.

Кредиторка відображається:

- по постачальниках;
- по статтях витрат;
- по датах оплати;
- у різних валютах.

4. Імпорт BAS/BAF може створювати прихід.
5. Для зброї імпорт також створює серійні одиниці.
